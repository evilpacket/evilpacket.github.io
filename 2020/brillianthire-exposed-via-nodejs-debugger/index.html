brillianthire.io Exposed via Node.js Debugger

Written by Adam Baldwin with ♥ on 2 August 2020 in 2 min

Last year I spent some time researching Internet exposed Node.js debuggers. As I went over the data many of the hosts didn’t have any information that could be used to identify who it belonged to. This made it very difficult to get in touch and disclose the issue for most exposed hosts. Yes I realize I could have dropped a little note on disk but that felt like a line I didn’t want to cross without explicit permission.

For those that made it obvious I reached out to hopefully save them the pain of an attacker getting ahold of it first.

One of those organizations I reported my findings to was brillianthire.io which is owned by SAP.

What I reported was an internet exposed host that had a node.js debugger running on it that provided access to.

  • Source code for the exposed application
  • Sensitive internal host names and other info
  • DB encryption key material
  • Production AWS access credentials with access to

Bounty? 😬

While I figured it was a long shot, I figured SAP might have a bounty program and I would ask because we all know the answer is no if you don’t ask and this was a signifigant exposure for SAP and Brilliant Hire customers.

The answer I received was surprising and confusing. 😳

We have invite-only bug bounty program on Bugcrowd. So unfortunately this report does not qualify for the bounty.

I could understand if they declined because it was out of scope but to say that the program was private so they wouldn’t provide an award.

BugCrowd did engage with me privately via email however their suggestion was to ask for an invite to the SAP private bounty program so I could recieve awards on future vulnerabilities. Pretty dispointing.

In the end SAP put my name on this page with zero explaination or disclosure - Acknowledgments to Security Researchers - Previous Months (2014 on wards)
